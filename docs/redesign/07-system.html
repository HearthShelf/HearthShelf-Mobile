Screen 07 · Sign-in

Sign-in

The hearth-art welcome is the app's best first impression — full-bleed ember artwork, the wordmark glowing out of the dark. That identity is kept and locked. What gets rebuilt is everything around it: the screen ignores the theme system entirely (hardcoded hex, no light mode), email sign-in is missing forgot-password and show-password, 2FA dead-ends with a raw status string, and the Terms line isn't tappable.

Teardown — what's wrong today

  • Hardcoded hex everywhere — the whole screen is static #bd863f/#f0e6d6/#0e0d0c; user accent and light mode are ignored (D-CONSIST names sign-in explicitly).
  • No forgot-password — email mode is email + password + Sign in. A mistyped password is a hard wall.
  • No show/hide password — long random passwords get typed blind.
  • 2FA dead-ends — accounts needing a second factor hit raw copy: "needs another step: {status}". No code entry exists.
  • Terms & Privacy aren't links — 10px static text "By continuing you agree to the Terms & Privacy Policy"; a compliance gap, not just a UX one.
  • Busy state is a bare ActivityIndicator swapped into the pressed button; the other providers stay full-strength and tappable-looking.
  • Errors are 13px centered text under the form — easy to miss on the dark art background.
  • No signup path note — OAuth-only signup is a product decision, but email mode never says so; users hunt for a Create-account button.
  • Auth block floats mid-screen — buttons sit ~78px below a centered hero, in the middle of the screen rather than the thumb zone; on a tall phone the primary actions are a stretch, and there's a lot of empty art between hero and buttons.

Redesign moves

  • The "sitting by the hearth" photo, bottom-aligned layout — the shipping firelit photograph fills the screen; the wordmark rides the upper third over it, and the entire auth block is anchored to the bottom (thumb zone). A bottom scrim keeps buttons and terms legible over the busiest part of the image. On keyboard open the block lifts above the keyboard so nothing is hidden.
  • Identity kept, locked — wordmark hero ("Hearth" gold regular + "Shelf" cream bold, Libre Baskerville). Never re-colored, never split. No flame glyph on this screen — the photo already has a real fire, so a drawn flame would double up.
  • Rebuilt on theme tokens — surfaces, text, and the accent on links/spinners come from the reactive theme. Dark is the framed default; a light-mode variant exists (art dims to a warm vignette).
  • Providers kept — Google (native credential sheet on Android, browser SSO elsewhere), Apple (iOS only), Discord (browser), email. Busy = inline spinner in the pressed button + all others dim to 40% and disable.
  • Email mode completed — show/hide password eye, "Forgot password?" (opens Clerk hosted reset), and an honest "New here? Sign up with Google, Apple, or Discord above" note.
  • 2FA designed — a real code-entry step (6 boxes, paste-aware, backup-code fallback) replaces the dead-end string.
  • Errors become a banner — destructive-wash card above the form with icon + plain-language message.
  • Terms & Privacy are tappable links — underlined, 12px, open in-app browser.

System · Boot splash

Splash phases — "the hearth ignites"

The splash is already the app's signature moment: anti-banding coal→ember gradient, 24 seeded ember particles on ballistic paths, pulsing halos, the flame flickering to life. The look is kept; the redesign work is twofold — the phases layered on top (connecting, select-server, error, no-servers) where today a raw server string can be the first thing a new user ever reads, and a performance pass on the crackling-fire animation so it stays 60fps on low-end Android during the worst moment to drop frames: cold boot.

Teardown — what's wrong today

  • Raw errors leak on a first-impression screen — friendlyError only catches a few known patterns; anything else prints the raw server string under the flame.
  • No-servers is a dead end — a signed-in user with zero linked servers gets a message and a Log out; nothing explains what a server is or how to get one linked.
  • No way to see detail when you need it — power users debugging a self-hosted box get the same softened copy with the real error hidden entirely (when friendlyError does match).
  • Static colors — acceptable for a boot screen that renders before the theme loads, but the accent on Retry should pick up the user's accent once known (D-CONSIST).
  • Select-server rows are bare — name + url, no indication of which was last used or which is default.
  • The crackle animation is 24 individually-animated SVG nodes — each ember is a react-native-svg <Ellipse> with a radial-gradient fill and its own withRepeat clock. SVG on RN re-renders per node; 24 of them + 2 gradient halos, running the instant the app cold-boots (JS still warming, GC churning), is exactly where cheap phones stutter. Motion is already worklet-driven (good) — the cost is node count and SVG rasterization, not the JS thread.

Redesign moves

  • Ember scene, same look, cheaper draw — the gradient ramp, seeded particles, pulsing halos, FlameLogo flicker + ignite, and wordmark all stay visually identical. The animation is optimized (see the box below) so it holds 60fps on low-end Android. Nothing added on top during connecting but one status line.
  • Friendly-first errors, details on demand — every error maps to one of ~6 plain-language patterns; a collapsible "Show details" row reveals the raw string + server url in mono for debugging. new
  • No-servers becomes an explainer — what a server is, plus a "How to link a server" help link into docs, alongside Manage servers and Log out. new
  • Server rows get context — "Last used" and "Default" chips; the default row sorts first.
  • Actions annotated and kept — Retry primary (accentGlow), Manage servers ghost, Log out text link, server row select.

Optimizing the crackling fire — keep the look, cut the cost

Same visual, cheaper to draw. The goal is 60fps on a low-end Android during cold boot, when the JS thread is busy and every dropped frame is visible. Ordered by payoff vs. effort — do 1–3 first; they're free wins that need no new dependency. 4 is the real fix if profiling still shows jank.

1 · one draw surfaceToday each ember is its own <Ellipse> node in react-native-svg (per-node rasterization + a radial-gradient fill each). Collapse the whole field into one host view: draw the embers as small Reanimated views with a pre-baked radial-glow PNG (a single cached texture, tinted per-ember via tintColor) instead of live SVG gradients. One texture uploaded once, N cheap transformed quads — no per-frame vector rasterization.
2 · animate transform + opacity onlyConfirm every ember worklet drives ONLY translateX/Y, scale, and opacity (all GPU-composited). Never animate width/height/radius or the gradient stops — those force layout/raster. The ballistic path should be position via transform, not a re-laid-out ellipse.
3 · right-size the field24 is a desktop-mockup number. Drop to ~10–14 on low-RAM devices (gate on Device.totalMemory / a coarse tier check) and keep 24 only on capable hardware. Perceptually the field reads the same; the raster/composite budget scales with node count. Overlap and motion-blur hide the lower count.
4 · one shared clockReplace 24 independent withRepeat timelines with one shared driver — a single useFrameCallback (or one master SharedValue ramp) that a useDerivedValue reads per ember, offsetting each by its seed. One clock ticking the field is far less overhead than 24, and it makes pause-on-blur trivial (see 6).
5 · Skia if you go furtherIf 1–4 aren't enough, the fire is a textbook @shopify/react-native-skia case: one <Canvas>, particles drawn in a single frame callback, GPU-composited, zero per-node RN views. Not installed today — worth it only if the fire becomes richer (real flame licks, embers reacting to touch). For the current look, 1–4 avoid adding a dependency.
6 · don't burn battery idleThe splash can sit on "connecting" for seconds on a slow server. Pause the field's clock when the app backgrounds (AppState) and drop to a slower tick after ~4s of connecting — a calm, slow crackle, not a full-rate particle sim, once the wow-moment has landed. Stop it entirely the instant the scene starts its exit fade.
7 · gradient ramp is fineThe anti-banding coal→ember background ramp and the two halos are cheap (static gradient + 2 opacity pulses) — leave them. The cost was never the background; it's the 24 live vector particles.

Net: steps 1–4 keep the exact look with a fraction of the draw cost and no new dependency — the same optimized ember system is what the sign-in screen reuses (a handful of particles over the hearth photo). Verify with the Perf Monitor / Android GPU profiler on a low-end device before and after.

System · Offline

Offline mode

The plumbing is genuinely good — a 4s Clerk timeout falls back to the cached session, the splash lifts, and Home rebuilds shelves from the downloaded catalog. The surface is where it falls short: the floating pill makes the whole thing look like one button while only "Retry" reads as tappable, and coming back online happens silently.

Teardown — what's wrong today

  • Ambiguous tap target — the OfflineBanner pill's entire surface triggers Retry, but visually only the word "Retry" looks tappable; users tap the text and think the rest is dead chrome (D-CONSIST: one obvious control per action).
  • Floating pill floats over content — it overlaps the Home header on scroll instead of owning a slot.
  • Reconnection is silent — the banner just disappears; users don't know their progress queue flushed or that sync happened.
  • Retry gives no feedback — no spinner, no "still offline" acknowledgment when the retry fails.

Redesign moves

  • Banner becomes a status chip bar — a full-width bar seated under the status bar (content starts below it, no overlap): cloud-off icon + "Offline — downloaded books only" + an explicit bordered Retry button. Only the button is tappable, and it looks it.
  • Retry acknowledges — button shows an inline spinner ~1s; on failure the bar text pulses "Still offline" before settling back.
  • Reconnected moment new — on network return the bar flips green: "Back online — syncing…" while the pending-progress queue flushes, then slides away.
  • Offline Home framed — downloaded shelves + skeleton-free cached render, per D-STATES (offline = cached content + status chip).

System · Feedback

The toast system

Today's toast is one global pill: check icon + text, 1900ms, centered in the mini-player band — and pointerEvents: none, so it can never host an action. That single limitation blocks D-FINISH across the whole app ("Finished · Edit date / Undo" has nowhere to live). The redesign makes the toast a three-variant system in the same spot.

Teardown — what's wrong today

  • pointerEvents: none — the toast cannot receive taps, so no Undo, no Edit date; every reversible action either grows a confirm dialog or ships irreversible (blocks D-FINISH).
  • Static colors — dark pill + hardcoded accent icon in every theme; in light mode it's a floating black slab, and the icon ignores the user's accent (D-CONSIST).
  • One shape for every message — confirmations, multi-item operations, and progress all get the same check-icon pill.
  • Single-instance replacement — a second toast stomps the first mid-read (acceptable, but timing must respect the action variant).

Redesign moves

  • Three variants, one position — confirm (icon + text, 1900ms), action (text + tappable accent actions, 4000ms), progress (mini spinner + text, dismisses on completion). Position kept: centered in the mini-player band, above the dock.
  • Action toast is tappable new — the enabler for D-FINISH's "Finished · Edit date / Undo" and every undoable operation (hide series, remove download, dismiss).
  • Reactive theme — pill surface, text, and accent all come from theme tokens; light mode gets a light pill with proper elevation.
  • Queue-aware replacement — a new toast replaces a confirm instantly but lets an action toast finish its 4000ms (or cut it short only for another action toast).

System · Entry & exit

Invite landing & exit-confirm

Two small moments that bookend a session. The invite deep link (app.hearthshelf.com/invite?token) currently shows a bare spinner + "Opening your invitation…" — functional, but it's a brand-new user's literal first screen, and a bad token has no designed failure. Exit-confirm (back-again-to-exit) works well and is kept as-is.

Teardown — what's wrong today

  • Invite landing is a bare spinner — no brand moment, no hint of what's being joined; the token routing underneath (stash → signed-in redeems at connect, signed-out persists through sign-in) is solid and kept.
  • No invite failure state — an expired or already-redeemed token has no designed screen; users see a generic error or a silent bounce.
  • Exit toast reuses the confirm pill — fine, but its 2s arm window was never spec'd against the toast system's timings.

Redesign moves

  • Invite becomes a hearth moment — flame + "Opening your invitation…", and when the token resolves, the server name reveals in a card before routing: joining something with a name feels like being welcomed, not processed.
  • Designed failure new — expired/used/invalid tokens get plain-language copy + "Ask for a new link" guidance + a path back to sign-in.
  • Exit-confirm kept — hardware back on Home arms "Press back again to exit" for 2s; now formally a confirm-variant toast with its own arm-window spec.
